# 데이터 유효성 검사

일부 소프트웨어는 Bluesky 또는 atproto 스키마 유효성 검사에 대해 크게 걱정할 필요가 없습니다. 예를 들어, 클라이언트 앱과 봇은 일반적으로 Bluesky API에서 받은 데이터가 유효하다고 가정할 수 있으며, 레코드를 생성하려 할 때 PDS가 스키마 유효성 검사를 다시 확인하는 책임을 집니다.

이벤트 스트림(파이어호스)에서 직접 데이터를 소비하는 소프트웨어는 더 주의해야 합니다. 또한 새로운 도구나 영리한 해킹을 설계하는 개발자들은 데이터 유효성 검사에 관한 몇 가지 기대사항과 엄격한 제한을 인식해야 합니다.

개별 레코드가 어떤 이유로든 유효성 검사에 실패하면, 해당 레코드 전체는 무시되어야 하지만 동일한 저장소의 다른 레코드는 처리되어야 합니다. 저장소 커밋 데이터나 저장소 구조("MST")에 문제가 있는 경우, 저장소를 무효로 표시해야 하지만 이전 콘텐츠를 인덱스에서 제거하거나 삭제할 필요는 없습니다. 새로운 유효한 커밋이 수신되는 즉시 저장소를 다시 유효로 표시해야 합니다. 이 로직은 주로 파이어호스에서 데이터를 소비할 수 있는 피드 생성기, AppView, 모더레이션 서비스에 적용됩니다.


## 권장 데이터 제한

다음은 비공식적인 가이드라인으로, 주로 대략적인 규모 기대치를 전달하기 위한 것이며 시간이 지남에 따라 변경될 수 있습니다. 이는 atproto 명세의 일부가 아니지만, 이 제한을 초과하려고 하면 현재 또는 미래에 상호운용성 문제가 발생할 수 있습니다.

**CBOR 레코드 크기:** 개별 레코드를 수십 KByte 이내로 유지하세요. 더 많은 데이터를 저장해야 하는 경우, 텍스트 데이터라도 블롭 사용을 고려하세요. 합리적인 최대 레코드 크기 제한(`MAX_CBOR_RECORD_SIZE`)은 1 MiByte(1 메비바이트, 즉 1,048,576 바이트)입니다. 2025년 8월 기준으로 `subscribeRepos` Lexicon은 `#commit` 메시지 블록 크기를 2,000,000 바이트(메비바이트가 아님)로 제한하므로, 커밋당 1 MiByte 레코드 하나만 생성하거나 업데이트할 수 있습니다.

이벤트 스트림(파이어호스) "프레임"은 여러 레코드로 구성될 수 있으므로, 일반적인 CBOR 파싱에는 4-5 MByte 수준의 더 큰 제한이 권장됩니다.

**JSON 레코드 크기:** CBOR 인코딩이 레코드의 "정규" 인코딩이므로, 해당 인코딩에만 집중하는 것이 합리적입니다. 그러나 때로는 JSON 인코딩 크기에도 제한을 두는 것이 좋습니다. 합리적인 제한(`MAX_JSON_RECORD_SIZE`)은 2 MiByte(2,097,152 바이트)입니다.

**일반 문자열 길이:** Lexicon에서 지정한 문자열 길이가 있는 것과 없는 것 모두 포함하여, 레코드 내 문자열에 대한 전체 길이 제한. 바이트(UTF-8 인코딩)로 측정됩니다. 최대 수십 KByte로 유지하려고 노력해야 합니다. 상한 제한(`MAX_RECORD_STRING_LEN`)으로는 전체 CBOR 레코드 크기 제한에만 의존하는 것이 합리적입니다. 특히 일부 초기 구현에서는 8 KByte(8192 바이트) 제한이 있었습니다.

**일반 `bytes` 길이:** 문자열 제한과 동일하지만 바이너리 데이터용(`MAX_RECORD_BYTES_LEN`). 전체 CBOR 레코드 크기 제한에 의존하도록 권장합니다. 전체 레코드 크기와 마찬가지로, 수십 KByte 이상이 필요한 경우 블롭 사용을 권장합니다.

**CID 바이너리 인코딩 크기:** 전체 제한(`MAX_CID_BYTES`)으로 100 바이트를 권장합니다.

**컨테이너 중첩 깊이:** 예를 들어, 배열 안의 배열 안의 맵 등 몇 개의 레이어인지. CBOR 또는 JSON 파싱 라이브러리가 제한을 지원하는 경우, 기본값이 아마도 적절할 것입니다. 합리적인 제한(`MAX_CBOR_NESTED_LEVELS`)은 32 레벨의 중첩입니다.

**컨테이너 요소 수:** 예를 들어, 맵의 키 수 또는 배열의 요소 수. CBOR 또는 JSON 파싱 라이브러리가 제한을 지원하는 경우, 기본값이 아마도 적절할 것입니다. 합리적인 제한(`MAX_CBOR_CONTAINER_LEN`)은 128 x 1024 = 131,072 요소입니다.

**객체 키 문자열 길이:** 예를 들어, 객체의 키에 허용되는 바이트(UTF-8 인코딩) 수. CBOR 또는 JSON 파싱 라이브러리가 제한을 지원하는 경우, 기본값이 아마도 적절할 것입니다. 합리적인 제한(`MAX_CBOR_OBJECT_KEY_LEN`)은 8 KByte(8192)입니다.

**정수:** [atproto 명세](https://atproto.com/specs/data-model#data-types)에서 언급된 것처럼, 정수 값을 "64비트 부동소수점 안전" 상태로 유지하는 것이 강력히 권장되는 모범 사례입니다. 즉, 53비트의 정밀도로 제한하는 것입니다. 이는 수치 정밀도 손실 없이 JavaScript와의 호환성을 보장합니다. 구체적인 값은 `MAX_SAFE_INTEGER: 9007199254740991`이고 `MIN_SAFE_INTEGER: -9007199254740991`입니다.


## 스키마 없이 유효성 검사

[atproto Lexicon 시스템](https://atproto.com/specs/lexicon)은 알려진 스키마에 대해 데이터를 유효성 검사하기 위한 데이터 모델 및 스키마 언어를 설명합니다. 그런데 스키마를 알 수 없거나 존재하지 않는 데이터를 처리하는 경우에는 어떻게 해야 할까요?

데이터 타입별 가이드라인은 다음과 같습니다:

- `integer`: 안전한 제한 내의 값이어야 합니다(제한 섹션에서 논의)
- `string`: 유효한 UTF-8 인코딩이어야 합니다. 특정 유니코드 정규화는 예상되거나 요구되지 않습니다. 빈 문자열은 허용되지만, 가능하면 nullable 또는 optional 필드를 활용하는 것이 좋습니다.
- `bytes`: 비어 있을 수 있습니다(길이 0)
- `cid-link`: 프로토콜 명세에서 논의된 것처럼: CIDv1이어야 합니다; CBOR 인코딩에서 멀티베이스는 raw(타입 `0x00`)여야 합니다; 멀티코덱은 `dag-cbor`(`0x71`) 또는 `raw`(`0x55`)여야 합니다; 그리고 멀티해시 타입 SHA-256이 권장됩니다(하지만 엄격하게 요구되지는 않음)
- `array`: Lexicon 유니온이 제공하는 유연성으로 인해 이질적인 타입의 요소를 가질 수 있습니다.
- `object`: 키는 모두 문자열이어야 하며, `string` 필드와 유사한 데이터 요건을 따릅니다. 빈 키 문자열은 허용되지 않습니다. `$`로 시작하는 키 이름은 프로토콜 사용을 위해 예약되어 있습니다(예: `$bytes` 및 `$type`)
- `blob`: `ref`는 유효한 `cid-link`여야 하며, `raw` 멀티코덱이어야 합니다(`dag-cbor`가 아님). `size`는 음수가 될 수 없습니다. `mimeType`은 빈 문자열이 될 수 없습니다. "레거시" 블롭 형식은 읽기에서 지원되어야 하지만, 생성되는 새 레코드는 일반 블롭 형식이어야 합니다. 추가 참조 타입(예: 대체 멀티코덱 또는 해시 타입)의 지원은 향후 프로토콜의 진화 및 확장 지점이 될 가능성이 높지만, 현재 버전의 저장소 및 데이터 모델은 좁게 정의되어 있습니다.
