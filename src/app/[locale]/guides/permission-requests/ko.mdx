이 페이지는 OAuth 플로우에서 권한 스코프를 **사용하는** 방법을 설명합니다. 커스텀 Lexicon을 구축하고 있으며 자체 권한 집합을 **정의하고 게시**하려는 경우, [권한 집합](/guides/permission-sets)을 참조하세요.

## 개요

이 가이드는 atproto 애플리케이션의 권한을 요청하고 정의하는 방법을 설명합니다. [OAuth](/guides/auth)가 인가를 처리하며, 이 페이지에서는 앱이 *무엇에* 접근할 수 있는지 지정하는 방법을 다룹니다.

atproto 리소스에 대한 제어는 [권한](/specs/permission)을 사용하여 부여됩니다. `repo:com.example.post`와 같은 개별 권한은 특정 레코드 유형 또는 API 엔드포인트에 대한 접근을 허가합니다. 관련 권한들은 사람이 읽을 수 있는 제목과 설명을 가진 **권한 집합**으로 묶을 수 있습니다. 이것이 사용자가 인가 대화 상자에서 보게 되는 것입니다.

**커스텀 Lexicon**을 사용하는 앱을 구축하는 경우, 그에 맞는 권한 집합을 정의해야 합니다. 이렇게 하면 불투명한 네임스페이스 ID 목록 대신, 앱이 무엇을 할 수 있는지에 대한 명확한 설명을 사용자에게 제공할 수 있습니다. 구조에 대해서는 [권한 집합 설계](/guides/permission-sets#permission-set-design)를 참조하세요.

**기존 Lexicon**(예: Bluesky의 `app.bsky.*`)만 사용하는 앱을 구축하는 경우, 게시된 권한 집합을 참조하거나 개별 세분화된 권한을 요청할 수 있습니다. 사용자에게 어떻게 표시되는지는 [OAuth 플로우 예시](#o-auth-flow-examples) 섹션을 참고하세요.

## 권한 요청하기

앱이 atproto 아이덴티티 제공자와 OAuth 플로우를 시작할 때, 인가 요청의 `scope` 파라미터에 포함하여 특정 권한 집합을 요청할 수 있습니다. OAuth 클라이언트를 등록할 때 [`oauth-client-metadata.json` 파일](/specs/oauth#clients)에서 *공백으로 구분된 목록*으로 제공되어야 합니다:

<OauthScopeMetadata />

AT Protocol의 OAuth는 [Pushed Authorization Requests](https://auth0.com/blog/what-are-oauth-push-authorization-requests-par/)（PAR）를 사용합니다. 클라이언트 메타데이터 파일은 세션 생명주기 동안 동적으로 가져오며, `oauth-client-metadata.json`의 `scope` 파라미터는 앱이 실제로 만드는 인가 HTTP 요청과 *일치해야 합니다*:

<OauthAuthUrl />

앱에 필요한 기능을 커버할 만큼의 권한만 요청하도록 노력해야 합니다. 다른 애플리케이션의 게시물을 포함하여 사용자의 AT 레코드를 생성, 업데이트 또는 삭제하는 광범위한 권한을 부여하는 [전환 스코프](/specs/oauth#authorization-scopes)도 제공하지만, 일반적으로 이러한 광범위한 권한 요청은 피해야 합니다. 세분화된 권한은 사용자의 신뢰를 구축하고 인가 프로세스 중 마찰을 최소화합니다.

사용자 경험 관점에서 요청하는 권한 집합의 수를 낮게 유지하는 것이 좋습니다. 권한 집합이 추가될수록 인가 대화 상자에 더 많은 정보가 표시되어 결정 피로로 이어질 수 있습니다. 사용자가 요청된 권한의 일부만 부여하도록 허용할 수 있지만, 이는 엣지 케이스로 이어질 수 있습니다.

## 권한 유형

PDS 인스턴스의 사용자 소유 리소스와 관련된 권한은 다섯 가지 유형으로 표현됩니다:

- `repo`: 공개 저장소 (레코드 및 컬렉션)
- `rpc`: 서비스 인증 (외부 서비스에 대한 API 호출)
- `blob`: 업로드된 미디어 파일
- `identity`: DID 및 핸들
- `account`: 호스팅 상태, 이메일 주소

와일드카드(*)는 스코프 문자열 구문에서 허용되며 모든 레코드에 대한 접근을 부여합니다. 가장 일반적인 예는 이미지와 동영상 작업을 위한 `blob:*/*`입니다. 부분 와일드카드는 *지원되지 않습니다*. 예를 들어, `app.bsky.*`는 유효한 스코프 문자열이 아닙니다. 특정 네임스페이스나 Lexicon의 여러 권한을 요청하려면 개별적으로 나열하거나 [권한 집합](/guides/permission-sets)을 사용해야 합니다. 이는 과도하게 광범위한 권한 요청을 방지하기 위한 의도적인 설계입니다.

권한 집합은 자체적으로 Lexicon 스키마이며 다른 Lexicon과 마찬가지로 네임스페이스로 구분됩니다. 다른 Lexicon과 달리 권한 집합은 스코프 문자열에서 `include:` 접두사로 시작하며 동적으로 업데이트될 수 있습니다.

## 클라이언트

OAuth 클라이언트 소프트웨어는 전역적으로 고유한 `client_id`로 식별됩니다. 동일한 클라이언트 소프트웨어의 서로 다른 변형은 서로 다른 `client_id` 값을 가질 수 있습니다. 예를 들어, 동일한 소프트웨어의 브라우저 앱과 Android(모바일 OS) 변형은 서로 다른 `client_id` 값을 가질 수 있습니다. `client_id`는 클라이언트 메타데이터 JSON 문서를 가져올 수 있는 완전히 정규화된 웹 URL이어야 합니다. 예를 들어, `https://app.example.com/oauth-client-metadata.json`. `client_id`에 대한 추가 정보:

- W3C URL 명세를 따르는 올바른 형식의 URL이어야 합니다
- 스키마는 `https://`이어야 하며 포트 번호가 포함되어서는 안 됩니다. 개발을 위한 `http://localhost` `client_id` 값에 대한 [특별 예외](/specs/oauth#localhost-client-development)가 있습니다.
- 경로에 `oauth-client-metadata.json`을 포함할 필요는 없지만 유용한 관례입니다.

<Note>
`oauth-client-metadata.json` 대신 `client-metadata.json`을 사용하고 있었다면, URL 문자열 대신 도메인이 인증 플로우 페이지에 표시되도록 이 변경을 적용할 수 있습니다.
</Note>

  <Image src={oauthdomain} alt="" className="w-full max-w-md mx-auto" />
</Container>

인증 플로우 페이지에는 부여되는 모든 관련 권한이 표시됩니다. 이는 [세분화된 권한](/specs/permission)을 사용하는 앱과 [전환용](/specs/oauth#authorization-scopes) OAuth 스코프를 사용하는 앱 모두에 해당됩니다.

클라이언트 앱은 인증 플로우 중에 실제로 승인된 인증 스코프를 반드시 확인해야 합니다. 특히, 계정 이메일 주소 읽기 권한(`account:email`)을 요청할 때, 사용자가 다른 권한을 부여하면서 해당 권한을 거부할 수 있습니다. 이론적으로, 전체 요청이 승인되는 동안 개별 권한이 거부될 수 있습니다.

## 변경 사항 롤아웃

새 권한 집합을 사용하도록 앱을 업데이트할 때 소프트웨어 릴리스와 권한 집합 업데이트를 조율하는 것이 중요합니다. 사용자 승인이 필요한 새 권한 집합을 추가하면 사용자가 이러한 새 권한을 부여하기 위해 앱을 다시 인가해야 합니다.

[클라이언트 메타데이터 문서](/specs/oauth#clients)의 릴리스 및 배포 프로세스는 앱의 코드 배포 프로세스와 별개일 수 있습니다. 그에 맞게 계획하세요. 새 권한을 요청하는 업데이트된 `oauth-client-metadata.json`은 해당 권한이 필요한 앱 버전보다 먼저 배포되어야 합니다.

권한 집합은 개정될 수 있지만 제거하거나 파괴적인 변경을 하면 기존 애플리케이션이 중단될 수 있습니다. 일반적으로 권한 집합은 설계된 애플리케이션과 밀접하게 연관되어야 하지만, 반드시 1:1 관계일 필요는 없습니다. atproto 애플리케이션은 여러 다른 Lexicon을 활용할 수 있다는 점을 기억하세요.

## OAuth 플로우 예시

atproto 아이덴티티 제공자를 *오직* 인증 목적으로만 사용하는 앱은 [`atproto` 권한만 요청하면 됩니다](/guides/permission-sets). 이는 다른 아이덴티티 제공자에서 기대할 수 있는 것과 동일한 OAuth 플로우를 제공합니다. Atmosphere 소셜 그래프의 기능은 사용되지 않으며 추가 권한이 필요하지 않습니다:

이것은 완전히 사용 가능한 Atmosphere 통합이며 많은 애플리케이션이 이것을 시작점으로 사용합니다. 간단하고 읽기 쉬운 "Authorize" 플로우를 생성한다는 점에 주목하세요. 요청하는 앱의 URL이 명확하게 표시되고, 권한 대화 상자에는 앱이 "당신을 고유하게 식별하려고 합니다"라고 명시되어 있지만 그 외에는 아무것도 없습니다.

다음으로 추가 권한을 요청하는 더 복잡한 통합을 생각해 보겠습니다. 기본적으로 요약 뷰로 표시됩니다:
