## HTTP 요청

웹 서버에서 사용자가 업로드한 파일을 임의로 제공하는 것은 다양한 콘텐츠 보안 문제를 야기합니다. 예를 들어, 동일한 "오리진"에서 제공되는 스크립트나 SVG 콘텐츠를 통한 크로스 사이트 스크립팅(XSS) 공격이 발생할 수 있습니다. **`getBlob`** 엔드포인트에 [Content Security Policy](https://developer.mozilla.org/en-US/docs/Web/HTTP/CSP)를 활성화하는 것은 사실상 필수이며, blob 저장소(**`getBlob`** 엔드포인트)에서 브라우저와 웹 애플리케이션에 직접 자산을 동적으로 제공하는 것은 지원되지 않습니다. 애플리케이션은 브라우저와 웹 에이전트에 제공하기 전에 독립적인 CDN, 프록시 또는 기타 웹 서비스를 통해 blob, 파일, 자산을 프록시해야 하며, 이러한 서비스는 보안 예방 조치를 구현해야 합니다.

이 엔드포인트에 대한 콘텐츠 보안 헤더 예시는 다음과 같습니다:

```jsx
Content-Security-Policy: default-src 'none'; sandbox
X-Content-Type-Options: nosniff
```

## 메타데이터

일부 미디어 타입에는 민감한 메타데이터가 포함될 수 있습니다. 예를 들어, JPEG 이미지 파일의 EXIF 메타데이터에는 GPS 좌표가 포함될 수 있습니다. 서버는 이러한 메타데이터의 의도치 않은 유출을 방지하기 위한 조치를 취해야 합니다. 예를 들어, blob을 저장하기 전에 해당 메타데이터를 제거하는 방법이 있습니다.

미디어 파일 파싱은 메모리 안전 버그와 보안 취약점의 잘 알려진 원인입니다. 콘텐츠 타입 감지("스니핑")조차도 취약점의 원인이 될 수 있습니다. 서버는 강력한 샌드박싱 메커니즘 없이 미디어 파일(이미지, 비디오, 오디오 또는 기타 비단순 형식)을 직접 파싱하지 않도록 강력히 권고됩니다. 특히, PDS 인스턴스 자체는 미디어 리사이징이나 트랜스코딩을 직접 구현해서는 안 되며, 이는 애플리케이션 레이어에서 처리되어야 합니다.

## 악의적 사용

더 풍부한 미디어 타입은 남용 및 불법 콘텐츠에 대한 위험성을 높입니다. 서비스는 이러한 콘텐츠가 감지 및 신고될 때 제거할 수 있는 적절한 메커니즘을 구현해야 합니다.

서버는 악의적인 리소스 소비를 방지하는 조치가 필요할 수 있습니다. 예를 들어, 의도적인 디스크 공간 고갈, 네트워크 혼잡, 대역폭 남용 등에 대비해 속도 제한, 크기 제한, 할당량 설정을 권장합니다.
