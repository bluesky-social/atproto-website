このページでは、カスタムLexiconのための権限セットを**定義・公開**する方法を説明します。OAuthフローで既存の権限スコープを**使用**したいだけの場合は、[権限リクエスト](/guides/permission-requests)を参照してください。

<Heading level={2} id="understanding-permission-sets">権限セットについて</Heading>

OAuthフローはセッションで付与されるアクセスのレベルを表す権限の「スコープ」を使用します。ユーザーのアカウントへの広範なアクセスを要求するのではなく、atprotoアプリは特定のレコードタイプやAPIエンドポイントに対して細かい権限を要求する必要があります。

コアとなるパターンはシンプルです。**アプリが必要とするコレクションへのアクセスのみを要求する**。例えば、アプリが独自のLexiconを使って投稿といいねを作成する場合、以下のようになります。

```
repo:com.example.post repo:com.example.like blob:*/*
```

これにより、アプリは`com.example.post`と`com.example.like`コレクションでレコードの作成/更新/削除を行う権限と、メディアアップロードの権限を得ることができます。ユーザーの認可ダイアログはこれらの特定の権限を明確に表示し、`transition:generic`のような広範なスコープが示す「ほぼすべてへのアクセス」という不穏な表現にはなりません。

よりわかりやすい認可UIにするために、関連する権限を人間向け説明付きの**権限セット**にまとめることができます。詳細は[権限セットの設計](#permission-set-design)を参照してください。

<Heading level={3} id="quick-start">クイックスタート</Heading>

独自のレコードタイプを作成するアプリに対する、最小限の`oauth-client-metadata.json`は次のようになります。

```json
{
  "client_id": "https://example.com/oauth-client-metadata.json",
  "client_name": "My Example App",
  "client_uri": "https://example.com",
  "redirect_uris": ["https://example.com/oauth/callback"],
  "grant_types": ["authorization_code", "refresh_token"],
  "response_types": ["code"],
  "scope": "atproto repo:com.example.post repo:com.example.like blob:*/*",
  "token_endpoint_auth_method": "none",
  "dpop_bound_access_tokens": true
}
```

これはアイデンティティ（`atproto`）、2つのレコードコレクションへの書き込みアクセス、メディアアップロードを要求します。完全なセットアップガイドについては[OAuthについて](/guides/about-oauth)を参照してください。

<Heading level={3} id="minimal-permissions">最小限の権限</Heading>

atproto OAuthを使用するアプリにおける究極に最小限（アイデンティティのみ、データアクセスなし）の権限：
- `atproto`

Blueskyのプロフィール情報を利用したい場合：
- `rpc:app.bsky.actor.getProfile?aud=did:web:api.bsky.app#bsky_appview`

画像や動画をアップロードする場合：
- `blob:*/*`

`transition:generic`のような粗い「トランジションスコープ」もまだサポートされていますが、細かな権限の採用を推奨します。トランジションスコープの概要については[認可スコープ](/specs/oauth#authorization-scopes)を、粒度の細かい権限仕様の完全版については[権限](/specs/permission)を参照してください。

<Heading level={2} id="permission-set-design">権限セットの設計</Heading>

フル機能のクライアントアプリは、多数の細かな権限（数十、場合によっては数百）を必要とします。権限管理を簡素化するために、Lexicon設計者は公開するスキーマの一部として権限の集合を定義できます。これらの権限セット自体もLexiconスキーマであり、`com.example.authBasicFeatures`のような名前空間IDで参照されます。

例えば、アプリケーションは`include:com.example.authBasicFeatures?aud=did:web:api.example.com%23svc_appview`のような権限セットを要求できます。これはアプリの動作に必要な権限の集まりを定義するLexiconに解決されます。

<PermissionSetLexicon />

`app.example.feed.authOnlyPost`や`com.example.authBasicFeatures`の例にある`auth`プレフィックスに注目してください。これは、そのLexiconが権限セットとして使われるよう設計されていることを示す命名規則です。

権限セットは、そのセット自身と同じNSID名前空間下のリソースを参照する権限のみを表現できます。例えば、`app.example.feed.authOnlyPost`というセットには、`app.example.feed.post`レコードへの権限や、リモートサービスへの`app.example.feed.getPostThread` APIエンドポイントのリクエスト権限を含めることができます。しかし、`app.example.actor.profile`への権限は付与できません。階層が一段上の`app.example.authFull`のような権限セットであれば、これらすべてのリソースへの権限を含められます。

`title`と`details`フィールドは説明用で、OAuthフローの一部として表示されます。これらの説明フィールドは国際化にも対応しています。詳細は[権限セット仕様](/specs/permission#permission-sets)を参照してください。

`rpc`権限における`inheritAud`の使用により、[継承の挙動](/specs/permission#rpc)を制御できます。

画像/動画の取り扱い用の`blob:*/*`権限は権限セットに含めることができず、常に個別に要求する必要があります。

権限セットもLexiconであり、同じツールチェーンを使って公開されます。`goat`を使用してLexiconを作成、リンティング、公開する手順については[Lexiconの公開](/guides/publishing-lexicons)を参照してください。

<Heading level={2} id="progressive-scope-requests">段階的なスコープ要求</Heading>

アプリはサインイン時にすべての権限を要求する必要はありません。最小限のスコープから始め、ユーザーが機能をオプトインするのに合わせて後から追加で要求できます。動的なスコープ選択や既存セッションのアップグレードなどの実装パターンについては[OAuthパターン集](/guides/oauth-patterns#progressive-scope-requests)を参照してください。
