## 앱 개발자를 위한 안내

이 가이드는 atproto 애플리케이션의 권한을 요청하는 방법을 설명합니다. [OAuth](/guides/auth)는 서드파티 애플리케이션이 사용자의 소셜 그래프에 접근하는 것을 인가하는 데 사용되며, 여기서 설명하는 개념들의 전제 조건입니다.

atproto 리소스에 대한 제어는 [권한(permissions)](/specs/permission)을 사용하여 설명하고 부여됩니다. 특정 Lexicon 네임스페이스(레코드 타입과 API 엔드포인트)와 관련된 권한 그룹은 "권한 세트(permission set)"로 묶을 수 있습니다. 두 개념 모두 [OAuth](/specs/auth)의 맥락에서 클라이언트 소프트웨어에게 PDS의 계정 리소스 접근 권한을 부여하는 데 사용됩니다: 예를 들어, 사용자의 공개 저장소에 특정 타입의 레코드를 쓰거나, 원격 서비스에 인증된 API 요청을 하는 능력이 이에 해당합니다. 개발자는 앱이 동작하는 데 필요한 권한을 선언하고, 사용자는 앱에 접근 권한을 부여할 때 해당 권한을 확인하게 됩니다.

일반적으로 더 깔끔한 UI를 위해 기존 권한 세트를 사용하는 것이 좋지만, 필요에 따라 개별 권한을 요청하거나 자체 권한 세트를 정의할 수도 있습니다. 이러한 요청이 OAuth 흐름에서 어떻게 표시되는지는 [OAuth 흐름 예시](#oauth-flow-examples) 섹션을 참고하세요.

## 권한 요청하기

앱이 atproto ID 제공자와 OAuth 흐름을 시작할 때, 인가 요청의 `scope` 파라미터에 특정 권한 세트를 포함하여 요청할 수 있습니다. OAuth 클라이언트를 등록할 때 [`oauth-client-metadata.json` 파일](/specs/oauth#clients)에 *공백으로 구분된 목록* 형태로 제공해야 합니다:

```
{
scope: "atproto rpc:app.bsky.feed.searchPosts?aud=* blob:*/*",
...
}
```

AT의 OAuth는 [Pushed Authorization Requests](https://auth0.com/blog/what-are-oauth-push-authorization-requests-par/) (PAR)를 사용합니다. 클라이언트 메타데이터 파일은 세션 수명 주기 동안 동적으로 가져오며, `oauth-client-metadata.json`의 `scope` 파라미터는 앱이 실제로 만드는 인가 HTTP 요청과 *일치해야* 합니다:

```
https://your-authorization-server.com/authorize?
    client_id=1234567890&
    redirect_uri=https://your-app/callback&
    scope=atproto rpc:app.bsky.feed.searchPosts?aud=* blob:*/*
    response_type=code&
    audience=https://myapi&
    state=1234567890
```

앱에 필요한 기능을 충족하는 최소한의 권한만 요청하도록 노력해야 합니다. [전환 스코프(Transition scopes)](/specs/oauth#authorization-scopes)는 다른 애플리케이션의 게시물을 포함하여 사용자의 AT 레코드를 생성, 업데이트, 삭제할 수 있는 권한을 부여하는 것처럼 보입니다. 일반적으로 이러한 광범위한 권한 요청은 피해야 합니다. 이는 사용자와의 신뢰를 쌓고, 인가 과정에서의 마찰을 최소화하는 데 도움이 됩니다.

사용자 경험 관점에서, 요청하는 권한 세트의 수를 적게 유지하는 것이 좋습니다. 권한 세트가 추가될수록 인가 대화상자에 표시되는 정보가 늘어나 결정 피로를 유발할 수 있습니다. 사용자가 요청된 권한의 일부만 승인하도록 허용할 수 있지만, 이는 예외적인 상황을 만들 수 있습니다.

## 권한 유형

PDS 인스턴스의 사용자 소유 리소스와 관련된 권한은 다섯 가지 유형으로 표현됩니다:

- `repo`: 공개 저장소 (레코드와 컬렉션)
- `rpc`: 서비스 인증 (외부 서비스에 대한 API 호출)
- `blob`: 업로드된 미디어 파일
- `identity`: DID와 handle
- `account`: 호스팅 상태, 이메일 주소

스코프 문자열 구문에서 와일드카드(*)를 사용할 수 있으며, 모든 레코드에 대한 접근을 허용합니다 — 가장 일반적인 예는 이미지와 비디오 작업을 위한 `blob:*/*`입니다. 부분 와일드카드는 *지원하지 않습니다* — 예를 들어, `app.bsky.*`는 유효한 스코프 문자열이 아닙니다. 특정 네임스페이스나 Lexicon에서 여러 권한을 요청하려면 각각 개별적으로 나열하거나 [권한 세트](#for-lexicon-designers)를 사용해야 합니다. 이는 지나치게 광범위한 권한 요청을 방지하기 위한 의도적인 설계입니다.

권한 세트는 그 자체로 Lexicon 스키마이며 다른 Lexicon처럼 네임스페이스가 지정됩니다. 다른 Lexicon과 달리, 권한 세트는 스코프 문자열에서 `include:` 접두사를 붙이며 동적으로 업데이트될 수 있습니다.

## 변경 사항 배포

새 권한 세트를 사용하도록 앱을 업데이트할 때, 소프트웨어 릴리스와 권한 세트 업데이트를 조율하는 것이 중요합니다. 사용자 승인이 필요한 새 권한 세트를 추가하면, 사용자는 새 권한을 부여하기 위해 앱을 재인가해야 합니다.

[클라이언트 메타데이터 문서](/specs/oauth#clients)의 릴리스 및 배포 프로세스는 앱 코드 배포 프로세스와 별개일 수 있습니다. 이에 따라 계획을 세우세요 — 새 권한을 요청하는 업데이트된 `oauth-client-metadata.json`은 해당 권한이 필요한 앱 버전보다 먼저 배포되어야 합니다.

## Lexicon 디자이너를 위한 안내

Lexicon은 JSON으로 작성됩니다 — [Lexicon 가이드](/guides/lexicon)에서 예시를 찾아볼 수 있습니다. Atmosphere HTTP API 메서드는 각각 특정 Lexicon 세트를 스코프로 하고 구현하므로, Lexicon은 적절한 쿼리 파라미터, 요청 본문, 응답 본문을 포함해야 합니다. 각 Lexicon은 권한 스코프가 됩니다 — 즉, `app.bsky.feed.post`는 AT 레코드 형식이자, 웹 엔드포인트이며, 세밀한 권한이기도 합니다.

Lexicon 스키마는 atproto Lexicon 저장소에 레코드로 공개적으로 게시됩니다. 기술적인 세부 사항은 [Lexicon 게시 및 해석](/specs/lexicon#lexicon-publication-and-resolution)을 참고하세요.

## 권한 세트 설계

완전한 기능을 갖춘 클라이언트 앱은 작동하기 위해 수십 또는 수백 개의 세밀한 권한이 필요합니다. 권한 관리를 단순화하기 위해, Lexicon 디자이너는 게시하는 스키마의 일부로 권한 "세트"를 정의할 수 있습니다. 이러한 권한 세트는 그 자체로 Lexicon 스키마이며, `com.example.authBasicFeatures`와 같은 네임스페이스 ID로 참조됩니다.

예를 들어, 애플리케이션이 `include:com.example.authBasicFeatures?aud=did:web:api.example.com%23svc_appview`와 같은 권한 세트를 요청할 수 있습니다. 이는 앱이 동작하는 데 필요한 권한 세트를 정의하는 Lexicon으로 해석됩니다:

```
{
  "lexicon": 1,
  "id": "com.example.authBasicFeatures",
  "defs": {
    "main": {
      "type": "permission-set",
      "title": "Basic App Functionality",
      "detail": "Creation of posts and interactions",
      "permissions": [
        {
          "type": "permission",
          "resource": "repo",
          "collection": ["app.example.post"]
        },
        {
          "type": "permission",
          "resource": "rpc",
          "inheritAud": true,
          "lxm": [
            "app.example.getFeed"
            "app.example.getProfile",
            ...
          ]
        },
      ]
    }
  }
}
```

`app.example.feed.authOnlyPost`와 `com.example.authBasicFeatures` 예시의 `auth` 접두사에 주목하세요. 이는 해당 Lexicon이 권한 세트로 사용하도록 설계되었음을 나타내는 명명 규칙입니다.

권한 세트는 세트 자체와 동일한 NSID 네임스페이스 하의 리소스를 참조하는 권한만 표현할 수 있습니다. 예를 들어, `app.example.feed.authOnlyPost` 세트는 `app.example.feed.post` 레코드에 대한 권한과 원격 서비스에 대한 `app.example.feed.getPostThread` API 엔드포인트 요청을 포함할 수 있습니다. 그러나 `app.example.actor.profile`에 대한 권한은 부여할 수 없습니다. 계층 구조에서 한 단계 위인 `app.example.authFull`과 같은 권한 세트는 이러한 모든 리소스에 대한 권한을 포함할 수 있습니다.

`title`과 `details` 필드는 설명적이며 OAuth 흐름의 일부로 표시됩니다; 예시는 [OAuth 흐름 예시](#oauth-flow-examples) 섹션을 참고하세요. 이러한 설명 필드의 국제화(i18n)가 지원됩니다 — [권한 세트 명세](/specs/permission#permission-sets)를 참고하세요.

`rpc` 권한의 `inheritAud` 사용은 [상속 동작을 제어](/specs/permission#rpc)할 수 있게 합니다.

이미지/비디오 처리를 위한 `blob:*/*` 권한은 권한 세트에 포함될 수 없으며 항상 별도로 요청해야 합니다.

## 린팅 및 검증

[`goat`](https://github.com/bluesky-social/goat)을 사용하여 Lexicon을 생성하고 게시할 수 있습니다. 게시 전에 `goat`으로 Lexicon을 린트하고 검증할 수도 있습니다. 예를 들어, `app.bsky.feed.post`를 가져와 린팅하는 방법:

```bash
$ goat lex pull app.bsky.feed.post
 🟢 app.bsky.feed.post

$ goat lex lint lexicons/app/bsky/feed/post.json
 🟡 lexicons/app/bsky/feed/post.json
    [unlimited-string]: no max length
    [unlimited-string]: no max length
error: linting issues detected
```

완벽한 것은 없죠 😊

`goat lex new record`로 새 Lexicon 레코드를 생성할 수 있습니다:

```bash
$ goat lex new record dev.project.thing
```

[스타일 가이드](/guides/lexicon-style-guide)에 따라 Lexicon을 개발한 후 `goat lex publish`로 게시합니다:

```bash
$ goat lex publish
 🟢 dev.project.thing
```

새 Lexicon을 게시한 후 `goat lex resolve`로 게시 여부를 확인할 수 있습니다:

```bash
$ goat lex resolve app.bsky.feed.post

{
  "$type": "com.atproto.lexicon.schema",
  "defs": {
    "entity": {
      "description": "Deprecated: use facets instead.",
      "properties": {
        "index": {
          "ref": "#textSlice",
          "type": "ref"
        },
        "type": {
          "description": "Expected values are 'mention' and 'link'.",
          "type": "string"
        },
        "value": {
          "type": "string"
        },
...
```

## 권한 세트 수정

권한 세트는 수정할 수 있지만, 제거하거나 파괴적인 변경을 하면 기존 애플리케이션이 손상될 수 있습니다. 일반적으로 권한 세트는 설계된 애플리케이션과 긴밀하게 연결되어야 하지만, 반드시 1대1일 필요는 없습니다. atproto 애플리케이션은 여러 다양한 Lexicon을 활용할 수 있음을 기억하세요.

## OAuth 흐름 예시

atproto ID 제공자를 *오직* 인증 목적으로만 사용하는 앱은 [`atproto` 권한만 요청하면 됩니다](/guides/scopes). 이는 다른 ID 제공자에서 기대할 수 있는 것과 동일한 OAuth 흐름을 제공합니다; Atmosphere 소셜 그래프의 기능을 사용하지 않으며 추가 권한이 필요하지 않습니다:

  <Image src={oauth} alt="" className="w-full max-w-md mx-auto"/>
</Container>

이는 완전히 사용 가능한 Atmosphere 통합이며, 많은 애플리케이션이 시작점으로 사용합니다. 단순하고 읽기 쉬운 "승인(Authorize)" 흐름이 생성된다는 점에 주목하세요. 요청하는 앱의 URL이 명확하게 표시되며, 권한 대화상자에는 앱이 "고유하게 식별하기를 원한다"고만 표시되고 그 이상은 없습니다.

다음으로, 추가 권한을 요청하는 더 복잡한 통합을 살펴보겠습니다. 기본적으로 이는 요약된 뷰로 표시됩니다:

**?** 아이콘을 클릭하면 각 권한 세트의 세부 정보를 볼 수 있습니다:
