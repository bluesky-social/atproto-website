## AT ProtocolにおけるOAuthの違い

Atprotoは[OAuth 2.1](https://www.ietf.org/archive/id/draft-ietf-oauth-v2-1-13.html)を基盤として、OAuth標準の特定の「プロファイル」を[仕様として定めています](/specs/oauth)。

他のOAuthシステムに慣れている場合、驚くかもしれない点がいくつかあります。

- **Atprotoは分散型**: 通常、アプリに「〇〇でサインイン」ボタンがある場合、ユーザーを認証できる機関（通常は大企業のいずれか）の選択肢が提供されます。atproto OAuthでは、アプリは認証プロバイダー（ユーザーの[PDS](/guides/the-at-stack#pds)）との事前の関係を持ちません。これは、事前に確立された関係を必要とするOIDCとatproto OAuthに互換性がない理由でもあります。

- **マイグレーション**: Atprotoユーザーは時間とともにサーバー（[PDS](/guides/glossary#pds-personal-data-server)）間でアカウントを移行できます。これを容易にするために、atprotoは柔軟な[アイデンティティ](/guides/identity)レイヤーを持ちます。このレイヤーにより、ユーザー名（[ハンドル](/specs/handle)）を静的なユーザーID（[DID](/specs/did)）に解決でき、さらにそこからユーザーのPDSを特定できます。ユーザーがアプリにログインする際、OAuthクライアントはこれらの関係を動的に解決します。

- **クライアントID**: 他のOAuthエコシステムでは、クライアントアプリがリソースサーバーに事前登録する必要があることが多いです。分散型システム（多数のクライアントと多数のリソースサーバーが存在する）ではこれは実用的ではないため、atprotoは[クライアントIDメタデータドキュメント](/specs/oauth#client-id-metadata-document)を使用してこの問題に対処します。

## DPoPの理解

[DPoP (Demonstrating Proof of Possession)](https://datatracker.ietf.org/doc/html/rfc9449)は、OAuthトークンをクライアントが保持する鍵に暗号学的に紐付けます。攻撃者がアクセストークンを傍受しても、秘密鍵なしでは使用できません。これは、長期セッション（最大2年）を持つ機密クライアントにとって特に重要です。

DPoPはatproto OAuthで必須です。**SDKを使用している場合**、自動的に処理されます。OAuthを手動で実装する場合（例：レコードを書き込むサイドカー）、知っておくべき重要な事項は以下の通りです：

- セッションごとに一度ES256鍵ペアを生成し、すべてのリクエストで再利用する
- すべてのリクエストには`DPoP`ヘッダーに新鮮なDPoP証明JWTが必要で、`Authorization: DPoP {access_token}`と組み合わせる
- 認可サーバーとPDS用のnonceを別々に追跡する（異なる場合がある）
- `error="use_dpop_nonce"`を伴う401を受け取った場合、`DPoP-Nonce`レスポンスヘッダーのnonceで再試行する
- PDSへのリクエストには、アクセストークンのSHA-256ハッシュを含む`ath`クレームを含める

[cookbook/python-oauth-web-app](https://github.com/bluesky-social/cookbook/tree/main/python-oauth-web-app)には`atproto_oauth.py`に完全な手動DPoP実装が含まれています。完全な仕様については、[OAuth仕様のDPoP](/specs/oauth#demonstrating-proof-of-possession-d-po-p)をご覧ください。

## トークンリフレッシュ

atprotoのOAuthアクセストークンには有効期限があります。トークンが期限切れになった場合、リフレッシュトークンを使用して新しいアクセストークンを取得する必要があります。SDKはこれを自動的に処理しますが、OAuthを手動で実装する場合：

- リフレッシュトークンをアクセストークンと一緒に安全に保存する
- APIコールが401で失敗した場合、再試行前にトークンリフレッシュを試みる
- 新しいアクセストークンは同じDPoP鍵ペアに紐付けられる

## セッションストレージ

BFFパターンを使用するウェブアプリケーションでは、OAuthセッションをユーザーセッションと関連付ける必要があります。一般的なアプローチには以下があります：

- **Cookieベースのセッション**: OAuthトークンをサーバーサイドに保存し、HTTP専用CookieのセッションIDをキーとして使用する
- **データベースバックのセッション**: ユーザーアカウントとともにトークンをデータベースに保存し、マルチデバイスセッションを可能にする

アーキテクチャパターンについては、[アプリの種類](/guides/about-oauth#types-of-app)をご覧ください。

## 段階的なスコープリクエスト

アプリケーションは常にサインイン時にすべての権限を必要とするわけではありません。[クライアントメタデータドキュメント](/specs/oauth#client-id-metadata-document)の`scope`パラメーターはアプリケーションが要求できるスコープの最大セットを宣言しますが、個々の認可フローはそのサブセットを要求できます。これにより、最小限のスコープから始めて、ユーザーがその機能を必要とする機能を選択するにつれて、後でより多くのスコープを要求することが可能になります。

### 認可前のスコープ選択

ユーザーによって異なるニーズを持つクライアントサイドアプリケーションの場合、OAuthリダイレクト前にユーザーが付与する[権限](/specs/permission)を選択できるようにするアプローチがあります。アプリケーションはユーザーの選択から[スコープ文字列](/specs/permission#scope-string-syntax)を構築し、[`authorize()`](https://github.com/bluesky-social/atproto/tree/main/packages/oauth/oauth-client#readme)に渡します：

```typescript
const BASE_SCOPES = ["atproto"];

// Request specific record collection access instead of broad permissions
const OPTIONAL_SCOPES = [
  // Access to your app's Lexicon records
  { id: "posts", scope: "repo:com.example.post", label: "Create posts" },
  { id: "likes", scope: "repo:com.example.like", label: "Like content" },
  // Or access to Bluesky profile data
  { id: "profile", scope: "rpc:app.bsky.actor.getProfile?aud=did:web:api.bsky.app#bsky_appview", label: "Read profile" },
  { id: "blob",  scope: "blob:*/*", label: "Upload images/video" },
];

function buildScopeString(selected: Set<string>): string {
  const granular = OPTIONAL_SCOPES
    .filter((s) => selected.has(s.id))
    .map((s) => s.scope);
  return [...BASE_SCOPES, ...granular].join(" ");
}

// pass the built scope string to the authorization flow
const url = await oauthClient.authorize(handle, {
  scope: buildScopeString(userSelections),
});
```

`repo:com.example.post`はそのコレクションへの完全な作成/更新/削除アクセスを付与することに注意してください。クエリパラメーターで特定のアクションに制限できます：`repo:com.example.post?action=create&action=update`。完全な構文については[権限仕様](/specs/permission#resource-types)をご覧ください。

コールバック後、アプリケーションはセッションで付与されたスコープを追跡し、それに応じて機能を表示または非表示にします。

### 既存セッションのスコープアップグレード

オプションの統合機能（アプリケーションの主要な[名前空間権限](/specs/permission#namespace-authority)外のスコープを必要とする機能）を持つ[BFFパターン](/guides/about-oauth#types-of-app)を使用するアプリケーションでは、ユーザーがすでにサインインした後にスコープのアップグレードが行えます。アプリケーションはユーザーの設定を保存し、拡張されたスコープセットで新しいOAuthフローを開始し、コールバックで古いセッションを置き換えます：

```typescript
// Your app's core permissions
const BASE_SCOPE =
  "atproto blob:*/* repo:com.example.post repo:com.example.like";

// Add permissions for another app's Lexicon (cross-namespace integration)
const EXTENDED_SCOPE =
  `${BASE_SCOPE} repo:social.othernet.feed.play repo:social.othernet.actor.status`;

// start a new OAuth flow with the expanded scope
const url = await oauthClient.authorize(handle, {
  scope: wantsIntegration ? EXTENDED_SCOPE : BASE_SCOPE,
});
```

サーバーはコールバックハンドラーが置き換えるべきセッションを識別できるよう、OAuthステートとともに古いセッション識別子を保存します。以降のログイン時に、サーバーはユーザーの保存された設定を参照し、拡張スコープを自動的に含めることができます。

[権限セット](/specs/permission#permission-sets)は明示的なスコープアップグレードの必要性を減らせることに注意してください：クライアントがトークンをリフレッシュする際、計算された権限が解決されたセットの変更を反映して更新される場合があります。新しい機能が既存の権限セットの[名前空間権限](/specs/permission#namespace-authority)内に収まる場合、公開されたセットを更新するだけで十分です。再認可は不要です。名前空間をまたぐ統合については、引き続き明示的なアップグレードが必要になります。スコープの変更をアプリケーションのリリースと調整するためのガイダンスについては、[変更のロールアウト](/guides/permission-requests#rolling-out-changes)をご覧ください。
