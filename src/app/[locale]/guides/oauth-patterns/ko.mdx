## 앱 유형

*가장 단순한* 유형의 앱은 [cookbook/vanillajs-oauth-web-app](https://github.com/bluesky-social/cookbook/tree/main/vanillajs-oauth-web-app#readme) 예시에서 보여주는 것처럼 OAuth 세션이 사용자 기기와 PDS 사이에서 직접 설정되는 것입니다. 전용 백엔드 서비스 없이 정적 리소스 호스팅만으로 가능하기 때문에 간단합니다.

앱에 장기 세션이 필요하지 않다면 이 방법으로 충분합니다. 하지만 다른 사용 사례에서는 앱이 "공개 클라이언트"이기 때문에 세션 수명이 제한됩니다. 이는 "기밀 클라이언트"와 대비되는 개념입니다. 기밀 클라이언트가 되려면 클라이언트 ID에 바인딩된 비밀 키를 설정해야 합니다. 클라이언트가 공개적으로 사용 가능하고 사용자 기기에서 실행된다면 클라이언트 비밀을 보호할 수 없습니다. 따라서 기밀 클라이언트를 구현하려면 백엔드 인프라 호스팅이 필요하며, 이는 [cookbook/go-oauth-web-app](https://github.com/bluesky-social/cookbook/tree/main/go-oauth-web-app#readme)과 [cookbook/python-oauth-web-app](https://github.com/bluesky-social/cookbook/tree/main/python-oauth-web-app#readme) 예시("BFF" 패턴 사용, 아래 참조)에서 보여줍니다.

기밀 클라이언트 대 공개 클라이언트에 대한 자세한 기술적 내용은 [클라이언트 유형](/specs/oauth#types-of-clients)을 참고하세요.

공개 클라이언트를 기밀 클라이언트로 "업그레이드"하는 데 사용할 수 있는 여러 설계 전략이 있습니다:

- [Backend For Frontend (BFF)](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-browser-based-apps#name-backend-for-frontend-bff): OAuth 세션이 앱 백엔드 서버와 PDS 사이에서 설정됩니다. 이 서버 사이드 세션은 세션 쿠키와 같은 메커니즘을 통해 사용자의 프론트엔드 세션과 연결됩니다. 프론트엔드는 백엔드를 통해 요청을 프록시하여 PDS에 요청을 보내며, 백엔드가 OAuth 인증을 처리합니다.
- [Token-Mediating Backend (TMB)](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-browser-based-apps#name-token-mediating-backend): BFF 패턴과 유사하지만, 세션이 설정된 후 백엔드가 OAuth 액세스 토큰을 프론트엔드에 전달하여 프론트엔드가 PDS에 직접 요청을 보낼 수 있게 합니다.
- [Client Assertion Backend](https://github.com/bluesky-social/proposals/tree/main/0010-client-assertion-backend): 단순화된 서버 사이드 로직을 갖춘 TMB 패턴의 대안으로 제안된 방식입니다.

이 세 가지 중 BFF 패턴이 현재 기밀 클라이언트 애플리케이션 구축에 권장되는 방법입니다. 이는 [cookbook/go-oauth-web-app](https://github.com/bluesky-social/cookbook/tree/main/go-oauth-web-app#readme)과 [cookbook/python-oauth-web-app](https://github.com/bluesky-social/cookbook/tree/main/python-oauth-web-app#readme) 예시에서 시연됩니다.

## DPoP 이해하기

[DPoP (Demonstrating Proof of Possession)](https://datatracker.ietf.org/doc/html/rfc9449)은 OAuth 토큰을 클라이언트가 보유한 키에 암호학적으로 바인딩합니다. 공격자가 액세스 토큰을 가로채더라도 개인 키 없이는 사용할 수 없습니다. 이는 장기 세션(최대 2년)을 가진 기밀 클라이언트에서 특히 중요합니다.

DPoP은 atproto OAuth에서 필수입니다. **SDK를 사용하는 경우** 자동으로 처리됩니다. OAuth를 수동으로 구현하는 경우(예: 레코드를 쓰는 사이드카), 알아야 할 핵심 사항은 다음과 같습니다:

- 세션당 ES256 키 쌍을 한 번 생성하고 모든 요청에 재사용합니다
- 모든 요청에는 `DPoP` 헤더에 새로운 DPoP 증명 JWT가 필요하며, `Authorization: DPoP {access_token}` 형식을 사용합니다
- 인증 서버와 PDS에 대한 논스를 별도로 추적합니다 — 다를 수 있습니다
- `error="use_dpop_nonce"`와 함께 401을 받으면 `DPoP-Nonce` 응답 헤더의 논스로 재시도합니다
- PDS 요청에는 액세스 토큰의 SHA-256 해시를 포함하는 `ath` 클레임을 포함합니다

[cookbook/python-oauth-web-app](https://github.com/bluesky-social/cookbook/tree/main/python-oauth-web-app)에는 `atproto_oauth.py`에 완전한 수동 DPoP 구현이 포함되어 있습니다. 전체 명세는 [OAuth 명세의 DPoP](/specs/oauth#demonstrating-proof-of-possession-d-po-p)을 참고하세요.
