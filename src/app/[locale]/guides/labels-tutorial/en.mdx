In this tutorial, you'll connect the Statusphere app from the previous tutorial to a labeler service running [Ozone](https://github.com/bluesky-social/ozone). Users will see moderator-issued labels alongside statuses in the feed, and will be able to report statuses to your labeler for review.

Nothing here is Bluesky-specific. Ozone is the same software the Bluesky moderation team uses, and the endpoints below are the protocol-level label endpoints any atproto labeler exposes.

## Prerequisites

This tutorial builds on the completed [Create a Social App](/guides/statusphere-tutorial) tutorial. You should have it running before starting here.

You'll also need a running Ozone instance that you can administer. Follow the [Ozone HOSTING guide](https://github.com/bluesky-social/ozone/blob/main/HOSTING.md) to stand one up. By the end of that setup you should have:

- A **labeler DID** (`did:web:...` or `did:plc:...`) with `#atproto_labeler` and `#atproto_label` entries in its DID document
- A **public service URL** that responds to `GET /xrpc/com.atproto.label.queryLabels?uriPatterns=*`
- An **admin login** for the Ozone UI

A quick way to confirm the labeler is reachable:

```bash
curl "https://<your-ozone-host>/xrpc/com.atproto.label.queryLabels?uriPatterns=*&limit=1"
# {"labels":[]}
```

## Part 2: Querying Labels

Labels live on the labeler service, not in any user's repo, so syncing them through Tap isn't the right path. You'll query the labeler directly each time the feed renders.

<S03 />

`com.atproto.label.queryLabels` is a public endpoint — no auth is required. The `uriPatterns` parameter accepts exact URIs or glob patterns; here you pass one exact URI per visible status.

The three helpers (`isHidden`, `isWarned`, `displayLabels`) interpret the values. `!hide` and `!warn` are system-level labels with prescribed UI behavior; anything else is a descriptive label like `spam` or `joke` that you can render however you want.

Open the Ozone UI, find the report in the queue, and apply a label — either a custom value like `spam`, or one of the system values `!warn` or `!hide`. Refresh the feed in your app and the label should appear (or hide the status entirely).

## Conclusion

You now have an atproto app that consumes signed labels from a labeler service and lets users submit reports to it. Querying on every render works for a small app, but production-shaped consumption uses the WebSocket stream at `com.atproto.label.subscribeLabels` to keep a local cache up to date — that's the next step from here.

Worth flagging that this setup didn't assume anything Bluesky-specific: the same Ozone instance, the same protocol endpoints, and the same client patterns work for any atproto app that wants community moderation.
